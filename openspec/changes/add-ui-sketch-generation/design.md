# Design — add-ui-sketch-generation

## Context

Auto-generate an Excalidraw wireframe at the tail of `/vector:raw` and `/vector:research`: when the
composed spec signals UI work, the command suggests generation; on confirmation, an embedded Sonnet
agent (`vector-ui-ux-designer`) emits a valid `.excalidraw` JSON that the binary validates and persists
under `.vector/specs/<id>/sketches/`; the board's `SpecDetailsDrawer` shows a download button; and the
existing `vector serve` SSE watcher live-updates with no restart. Opt-in per run, globally opt-out-able,
soft-failing. Source spec: `.vector/specs/add-ui-sketch-generation/spec.md`.

## Key decisions (locked)

- **Embedded agent, no global-skill/MCP dependency.** `vector-ui-ux-designer` lives in `kit/agents/`
  and is vendored into the binary; it never assumes `~/.claude/` or an Excalidraw MCP. Rationale:
  `architecture/distribution-packaging.md` requires every agent a `/vector:*` command invokes to be
  embedded.
- **`.excalidraw` download-only, no render.** The board shows a download button, not a thumbnail or
  inline preview. Rationale: in-binary render needs a JS runtime (`cli/go.mod` is stdlib-only); inline
  render needs `@excalidraw/excalidraw` (~2–3 MB), breaking `web/CLAUDE.md`'s light-bundle rule.
- **Async-in-session + SSE, no true daemon.** The sketch is generated by an async subagent inside the
  Claude session; on completion it calls the binary; the `vector serve` watcher propagates the change.
  Rationale: a binary daemon would force the binary to make its own LLM calls — today it makes zero
  (`route.go`, `summarize.go` only compute the meter cost).
- **Storage under `.vector/specs/<id>/sketches/<name>.excalidraw`.** Rationale: follows the
  shard-per-spec pattern; the prefix is already in `verifyArtifactPath`'s allowed-prefixes.
- **Additive schema, no `state.SchemaVersion` bump.** `Sketches []SketchRef` is `omitempty`, like
  `QuickWin bool`. Rationale: a legacy `SpecState` deserializes with `Sketches == nil`; no migration.
- **Sonnet for `vector-ui-ux-designer`.** Rationale: synthesizing visual layout/hierarchy from spec
  requirements is real reasoning, not mechanical text — justifies Sonnet per `product/token-routing.md`.
- **Hybrid heuristic + confirmation + global opt-out.** The heuristic evaluates the composed spec at
  the tail and prompts only on a strong UI signal; `--no-sketch` and `sketchEnabled: false` suppress
  the prompt. Rationale: sketch generation is optional and Sonnet-costly — forcing it breaks
  `product/token-routing.md`; the dev decides.
- **JSON output validation before persisting.** The binary verifies `{type, version, elements}` before
  copying; malformed output → silent soft-failure. Rationale: persisting a broken `.excalidraw` would
  mislead the board and anyone downloading it.

## Architecture

Pattern: **async-in-session + SSE push + CLI-owns-writes.** Heuristic + orchestration live in the
command's main loop (cheap). Sketch generation is delegated to a Sonnet subagent that runs async inside
the Claude session; on completion it calls the binary subcommand; the `vector serve` watcher propagates
the change over the existing SSE.

Layers affected:
- application/orchestration (kit): yes — tail step in `raw.md` (step 12) and `research.md` (step 15):
  detect + confirm + async spawn + routing.
- domain/agent (kit): yes — NEW `kit/agents/vector-ui-ux-designer.md` (Sonnet) emits the `.excalidraw`.
- data/state (Go): yes — `SketchRef` + `Sketches`, `Store.AttachSketch`, `"sketch"` in
  `artifactRelPath`.
- board (Go): yes — `Sketches` on `Card`; `"sketch"` in `validArtifact`; conditional Content-Type in
  `handleFile`.
- CLI (Go): yes — NEW `sketch.go` (`runSpecAttachSketch`) + dispatch in `runSpec` (`main.go`).
- config (Go): yes — `SketchEnabled *bool` + `IsSketchEnabled()`.
- presentation (web): yes — `board.ts` types, `useFileContent.ts` `ArtifactKey`, `entries.ts`,
  `SpecArtifactBrowser.tsx` (native `<a download>`). No board-card badge in V1.
- shared/scaffold: yes — embedded assets regenerated by `go generate`.

## Flow

1. Dev runs `/vector:raw` (or `/vector:research`); the draft card is registered via `vector spec create`.
2. At the tail, the heuristic evaluates the composed spec: strong signal iff §12 "Estados de UI" is
   non-empty **or** ≥2 layer keywords appear in title+body. Weak/absent → skip silently.
3. Strong signal and no opt-out (`--no-sketch` absent and `config.SketchEnabled != false`) → ask via
   `AskUserQuestion`.
4. Decline → command ends cleanly; spec stays a draft with no sketch.
5. Confirm → register the estimated routing (step 10) and spawn `vector-ui-ux-designer` async, then
   return immediately (the draft is already on the board).
6. The agent (Sonnet) reads the composed spec and `Write`s the `.excalidraw` JSON to a temp path
   (`.vector/tmp/<id>/sketch.excalidraw`).
7. The agent calls `vector spec attach-sketch <id> --file <temp>` with `Bash`. The binary validates the
   JSON shape (`{type, version, elements}`), copies to `.vector/specs/<id>/sketches/<name>.excalidraw`,
   and updates `state.json` via `Store.AttachSketch`.
8. The `vector serve` watcher (polling `.vector/`, default 1000 ms) detects the `state.json` change,
   calls `Broadcast()` → SSE → the board live-updates.
9. The `SpecDetailsDrawer` shows the download link for the new sketch.
10. Routing is registered at spawn time (step 5) with **estimated** tokens (the command does not wait
    for the agent): `vector spec route <id> --model sonnet --baseline opus --task "generate ui sketch"
    --tokens-in <est> --tokens-out <est> --precision estimated`.

## CLI contract (source of truth for writes)

- `vector spec attach-sketch <id> --file <path> [--name <name>] [--repo-root <dir>] [--json]` →
  validate JSON + `{type, version, elements}`, sanitize `--name`, copy the file, update `state.json`
  via `Store.AttachSketch`. Sole writer of the sketch artifact + state.
- `vector spec route <id> --model sonnet --baseline opus --task "generate ui sketch" --tokens-in N
  --tokens-out M --precision estimated` → register the agent routing for the meter.
- `GET /api/file?spec=<id>&artifact=sketch` → 200 with the `.excalidraw` bytes,
  `Content-Type: application/octet-stream`, `Content-Disposition: attachment; filename="<name>.excalidraw"`;
  404 when the spec has no sketch; 400 on unknown artifact key. V1 serves `spec.Sketches[0]`.
- `GET /api/board` → `Card.sketches` is an array of `SketchRef` (`{name, createdAt}`), omitempty. No new
  write endpoints (the write is exclusively `vector spec attach-sketch`, never HTTP).

## Open questions

- Inline render / thumbnail: unblock condition requires evaluating `@excalidraw/excalidraw` vs a custom
  renderer vs agent-side SVG export without breaking the light-bundle rule.
- True background daemon/worker: async-in-session degrades if the Claude session closes before the
  agent finishes; a decoupled model would require the binary to make its own LLM calls (out of scope
  today).
- UI-heuristic precision: the V1 rule (§12 non-empty **or** ≥2 keywords) is implementable; fine tuning
  (weighted scoring vs binary count, keyword-list adjustment after real false positives) is future work.
- Sketch version history: today multiple sketches are independent entries in `Sketches`; a versioning
  mechanism (v1, v2, …) with comparison is an open question.
- Dark-mode theming of the `.excalidraw` (`appState.theme` accepts `"light"`/`"dark"`) if the sketch is
  ever rendered.
- URL disambiguation for multiple sketches: `?artifact=sketch` serves the first; add `?name=<filename>`
  or a composite `ArtifactKey`?
- `board.SchemaVersion` bump: `Sketches []SketchRef` on `Card` is additive/omitempty and the frontend
  treats it as optional — does the board contract (`SchemaVersion = 2`) need a bump, or stay at 2?
