# Design — add-vector-quick-command

## Context

`/vector:quick` is the Vector-native mirror of the global `/quick-win` skill: an orchestrating project
command + a cheap Haiku refiner + direct implementation in the main loop + binary-owned state writes.
Unlike `/vector:raw`, it applies the change in the **same run** (no OpenSpec change, no Sonnet
validator) and the card lands in `review`. The durable product value is that a trivial change leaves a
board trail (`quickWin` + `work.logged`) inside Vector's single source of truth, instead of a parallel
`quick-wins.md` log. Source spec: `.vector/specs/add-vector-quick-command/spec.md`.

## Key decisions (locked with the user)

- **Apply-in-run lifecycle = `in-progress → review`.** The card is born `in-progress` (marked
  quick-win), the change is applied, `worklog` is recorded, and the card moves to `review`. Closing is
  explicit (`/vector:close`). Rationale: direct user answer; consistent with `/vector:apply` never
  auto-closing.
- **Light brief, no Sonnet validator.** The quick-win authors a short brief (not a 20-section spec);
  its "validation" is the repo's lint/typecheck gate. Rationale: preserves `/quick-win`'s lightness;
  the 20-section validator is disproportionate.
- **`quickWin` (bool) state marker + card badge.** Persisted on `SpecState`, exposed on board/API,
  shown on `SpecCard`. Rationale: the user explicitly asked the card to "be marked" and "show it is a
  quick-win".
- **Optional commit, asked every run.** Neither auto-commit (like `/quick-win`) nor never (like
  `/vector:apply`), but `AskUserQuestion`. Rationale: direct user answer.
- **Optional ticket/spec link reusing existing machinery.** `detectTicket`/`--ticket` for tickets and
  `--related kind=spec` for another spec id; only if it resolves, never blocking creation. Rationale:
  the user asked for it and the machinery already exists.
- **Board + `activity.jsonl` is the record; no parallel docs log.** Unlike `/quick-win`, no
  `quick-wins.md`/CHANGELOG in the user's repo. Rationale: single source of truth
  (`architecture/state-model.md`).
- **Own embedded refiner `vector-quick-refiner` (Haiku)**, not reuse of the global. Rationale: every
  kit agent is vendored/embedded except OpenSpec's own (`architecture/distribution-packaging.md`).
- **Escalate, do not expand.** A large/cross-cutting change → recommend `/vector:raw`; never grows
  silently. Rationale: `/quick-win`'s "Stay small".
- **Token routing.** Sanity-check/parse/link-resolution/implementation = main loop; refine = Haiku;
  no validator (`product/token-routing.md`).

## Architecture

Orchestrating project command (Vector-native mirror of `/quick-win`) + Haiku refiner delegation +
direct implementation in the main loop + state writes via the binary. The command coordinates,
refines (Haiku), implements, validates with the repo gate, logs the work, and transitions; the binary
is the sole writer of state (card, `quickWin`, ticket/related, worklog, status).

Layers affected:
- presentation (web): yes — `SpecCard` shows the read-only `quickWin` badge.
- application/orchestration (kit): yes — NEW `kit/commands/vector/quick.md`.
- domain/agent (kit): yes — NEW `kit/agents/vector-quick-refiner.md` (Haiku).
- data/state (Go): yes — `SpecState.QuickWin` + `CreateSpecParams.QuickWin`, `--quick-win` flag, `Card`
  projection.
- shared/scaffold: yes — embedded assets regenerated by `go generate`.

## Flow

1. User runs `/vector:quick "<description>" [{ticket|spec-id}]`.
2. Parse description (`RAW_QW`) from the optional arg; empty → `AskUserQuestion`, stop.
3. Confirm init: read `.vector/config.json` for `specPath`; missing → tell the user to run
   `vector init`.
4. Sanity-check (is this a quick-win): red flags (new screen/page/modal → `/vector:raw`;
   "broken"/"regression" → `/vector:bug`; multiple unrelated changes → ask to split;
   schema/migration/new endpoint → `/vector:raw` unless a literal one-line index) → recommend and
   stop; do not invoke the refiner.
5. Refine (Haiku): invoke `vector-quick-refiner` with `RAW_QW`; returns the light brief. Scope-guard:
   >~6 files, visible behavior change, or >3 blocking questions → escalate to `/vector:raw` and stop.
6. Clarify ≤3 blocking questions with `AskUserQuestion`; fold answers into the brief.
7. Resolve optional link (arg or `detectTicket` over `RAW_QW`): ticket → `--ticket`; existing spec id
   (`vector spec list --json`) → `--related '[{"kind":"spec",…}]'`. Ambiguous → ask or omit; never
   guess; never block creation.
8. Register the card `in-progress` + quick-win via the binary, passing the brief as the doc:
   `vector spec create --title … --id <slug> --status in-progress --quick-win [--ticket …]
   [--related …] --body-file -`.
9. Implement the change (main loop) with Read/Edit/Write, strict to `files-to-touch`. Growing out of
   scope → `git restore` the touched files, `vector spec status <id> needs-attention --reason "out of
   scope: use /vector:raw"`, stop.
10. Validate with the minimal gate of the detected stack (typescript→typecheck; go→`go vet`;
    python→`ruff`/`mypy`; rust→`cargo check`), scoped to the change. Do not run the whole suite.
11. Log the work: `vector spec worklog <id> --files … --tasks … --note …` → `work.logged`.
12. Commit (asking): `AskUserQuestion` "commit this quick-win?". Yes → atomic Conventional Commit in
    English, staging only the touched files + the doc; no `--no-verify`/`--amend`. No → leave the tree
    and report uncommitted changes.
13. Final transition: `vector spec status <id> review` (in-progress → review).
14. Token routing: `vector spec route <id> --model haiku --baseline opus --task "refine quick-win"
    --tokens-in N --tokens-out M` (only the refiner is routed).
15. Post-action summary (reuses the existing pipeline like `/vector:apply` §7).
16. Report: id, `quickWin`, transition, ticket/related, commit SHA or "uncommitted", gate result, and
    the next step (`/vector:close <id>`).

## CLI contract (source of truth for writes)

- `vector spec create --title … --id <slug> --status in-progress --quick-win [--ticket '<json>']
  [--related '<json-array>'] --body-file -` → create the card `in-progress`, mark `quickWin`, write
  the doc, seed ticket/related.
- `vector spec worklog <id> --files … --tasks … --note …` → `work.logged`.
- `vector spec status <id> review` → `in-progress → review`.
- `vector spec list --json` → resolve the arg as a spec id for the link.
- `vector spec route …` / `vector spec summarize …` → routing + post-action summary (reuse).
- `GET /api/board` → now includes read-only `quickWin` per card. No new write endpoints.

## Open questions

- Unified card-type taxonomy (`kind: feature|bug|quick-win`) vs the `quickWin` bool? V1 uses the bool;
  migrating to an enum is backward-compatible later.
- Should the badge be filterable/groupable on the board (a "quick-wins only" view) or only shown? V1
  only shows it.
- Should the optional arg accept both a ticket and a spec id at once, or only one? V1 assumes one.
- Should `/vector:quick` offer a direct close (`review → closed`) shortcut? V1 leaves closing explicit.
- Should the "≤~6 files" scope-guard be configurable in `.vector/config.json`? V1 keeps it fixed.
